# Requirements: Mathematical Proof CLI

**Archived milestone:** v1.2
**Archived on:** 2026-04-30

## v1 Requirements

All milestone requirements were satisfied and archived from the v1.2 delivery.

### Command Routing

- [x] **CMD-01**: Codex can recognize `$proof ...` commands through a stable, first-class routing entry point
- [x] **CMD-02**: The routing layer preserves subcommands and argument text accurately enough to invoke the intended Proof CLI command
- [x] **CMD-03**: Read-only commands such as status, theorem list, obligation list, blocker list, and search can execute directly from Codex
- [x] **CMD-04**: Repo-root resolution is deterministic for both project-local and global Codex entry points

### Mutation Workflows

- [x] **MUT-01**: Mutating commands such as theorem add, obligation add, blocker add, and snapshot go through the Proof CLI rather than manual state edits
- [x] **MUT-02**: When a mutating command is underspecified, Codex asks only for the minimum missing detail needed to continue
- [x] **MUT-03**: The command layer distinguishes read-only and mutating flows clearly enough to keep the trust boundary explicit

### Bootstrap and Fallback

- [x] **BST-01**: If the `proof` executable is unavailable, the command layer can detect that state and explain the next action clearly
- [x] **BST-02**: Global Codex skill configuration and project-local configuration can coexist without ambiguous routing behavior

### End-to-End Validation

- [x] **VAL-01**: A user can initialize or open a proof workspace and inspect it from Codex through the command-routing layer
- [x] **VAL-02**: A user can start a small theorem workflow from Codex, including theorem creation, obligation tracking, blocker tracking, and snapshotting
- [x] **VAL-03**: The routing layer is covered by smoke tests or equivalent verification that exercises real Codex-facing command paths
