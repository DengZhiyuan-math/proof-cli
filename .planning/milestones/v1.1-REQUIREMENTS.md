# Requirements: Mathematical Proof CLI

**Archived milestone:** v1.1
**Archived on:** 2026-04-29

## v1 Requirements

All milestone requirements were satisfied and archived from the v1.1 delivery.

### Collaboration Infrastructure

- [x] **COL-01**: User can identify contributors, roles, and authorship for theorem contracts, proof steps, obligations, blockers, bug reports, and repairs
- [x] **COL-02**: User can track provenance and reviewer identity for changes to shared proof objects
- [x] **COL-03**: User can propose, review, approve, reject, supersede, and dispute shared proof objects through explicit governance states
- [x] **COL-04**: User can attach comments and review threads to proof objects without mutating theorem truth state
- [x] **COL-05**: User can branch alternative theorem contracts or proof routes and compare them structurally
- [x] **COL-06**: User can publish and reuse team-scoped proof assets and domain packs with reviewable provenance
- [x] **COL-07**: User can export and import proof project state, shared dependencies, and review history without losing trust context
- [x] **COL-08**: User can enforce collaboration-aware policy rules for proposing, approving, publishing, callable state, machine-check acceptance, and dispute resolution
- [x] **COL-09**: User can inspect project summaries that surface pending reviews, disputes, branches, and collaboration state from the CLI
- [x] **COL-10**: The collaboration workflow can be validated on a real two-user proof project with at least one dispute, one shared imported theorem family, one reusable team asset, and one successful handoff

### Publication Pipelines

- [x] **PUB-01**: The project can map theorem-like objects into publication-oriented views
- [x] **PUB-02**: The project can mark claims as internal-draft, collaborator-ready, supplement-ready, paper-ready, disputed, blocked, withdrawn, or superseded
- [x] **PUB-03**: The project can export paper, supplement, and reproducible bundle outputs
- [x] **PUB-04**: The project can export citation provenance, review summaries, verification summaries, and governed release history without leaking internal-only material
- [x] **PUB-05**: A real mathematical section can move from proof workspace to publication package with preserved trust boundaries
- [x] **PUB-06**: Publication release history remains auditable across export and exchange boundaries
- [x] **PUB-07**: Internal-only material stays suppressed in paper-facing output
- [x] **PUB-08**: Publication validation exercises a real mixed-provenance slice
- [x] **PUB-09**: Supplement-facing and bundle outputs preserve trust distinctions
- [x] **PUB-10**: Bundle round-tripping preserves provenance and release history

### Retrieval and Snapshots

- [x] **PRET-01**: `proof retrieve` checks current theorem state, linked obligations, linked blockers, related memory artifacts, and nearby graph structure before loose text matching
- [x] **PRET-02**: `proof retrieve` reports what is already known in the current project, nearby attempts or insights, and structurally adjacent blockers
- [x] **PRET-03**: `proof project analyze` identifies the current bottleneck, central open obligations, failed routes that should not be retried blindly, and promising local next steps
- [x] **PSNP-01**: Snapshots capture the active theorem, open obligations, blockers, recent memory artifacts, relevant nearby graph nodes, and suggested next focus areas

### Validation

- [x] **PVAL-01**: The unfinished higher-rank radial system section can be modeled with the main target theorem, the Jacquet compression gap, the scalar-to-vector lifting problem, known lemmas and propositions, failed or incomplete proof routes, and an explicit graph showing the missing bridge
- [x] **PVAL-02**: The new memory and graph workflow materially helps section-level or project-level work, not just isolated theorem-local work
