# Milestone v1.3: Global Proof Plugin Integration

**Status:** ✅ SHIPPED 2026-04-30
**Phases:** 14-17
**Total Plans:** 4

## Overview

v1.3 turned the Codex-facing Proof CLI work from a repo-local wrapper experiment into a globally installable plugin-backed integration. The milestone added home-local plugin packaging, expanded the MCP tool surface, clarified the entry hierarchy, and validated the installed plugin copy end to end.

## Phases

### Phase 14: Global Plugin Packaging

**Goal**: Turn the repo-local Proof Routing experiment into a home-local plugin Codex can install and discover consistently.
**Depends on**: Phase 13
**Plans**: 1 plan

Plans:

- [x] 14-01: home-local plugin installation path, plugin-relative MCP wiring, and packaging validation

**Details:**
Phase 14 established `~/plugins/proof-routing` and `~/.agents/plugins/marketplace.json` as the canonical home-local footprint, added a deterministic installer, and made MCP script paths valid after relocation.

### Phase 15: Tool-Backed Proof Surface

**Goal**: Make the plugin expose a practical proof workspace tool surface instead of a manifest-only shell.
**Depends on**: Phase 14
**Plans**: 1 plan

Plans:

- [x] 15-01: expanded plugin MCP tools for proof inspection and core mutations

**Details:**
Phase 15 expanded the plugin tool surface to cover theorem show, obligation list, blocker list, search, and project analysis while preserving `proof codex` as the execution boundary.

### Phase 16: Activation and Entry Unification

**Goal**: Make the global plugin path understandable and non-ambiguous for everyday use.
**Depends on**: Phases 14 and 15
**Plans**: 1 plan

Plans:

- [x] 16-01: install/enable documentation and unified entry hierarchy across plugin, wrapper, and skills

**Details:**
Phase 16 documented the canonical entry order as plugin tools first, `proof codex ...` second, and `$proof ...` as compatibility only, while making repo-local skills explicitly debugging-scoped.

### Phase 17: Global Plugin E2E Validation

**Goal**: Prove the plugin-backed workflow is installable and testable by a real user.
**Depends on**: Phases 14, 15, and 16
**Plans**: 1 plan

Plans:

- [x] 17-01: installed-copy plugin validation through doctor and status workflow

**Details:**
Phase 17 validated the installed plugin copy directly by installing into a temporary home, loading the installed MCP server script, and exercising wrapper-backed doctor/status behavior.

## Milestone Summary

**Key Decisions:**

- Treat the plugin-backed MCP surface as the canonical Codex path once installed.
- Keep `proof codex ...` as the explicit CLI fallback and debugging surface.
- Keep Proof CLI as the source of truth rather than moving proof-state semantics into the plugin.
- Validate the installed plugin copy directly instead of assuming repo-local behavior generalizes automatically.

**Issues Resolved:**

- The plugin now has a deterministic home-local installation path.
- The global plugin tool surface now covers routine proof inspection and core mutation workflows.
- The relationship between plugin tools, `proof codex ...`, and `$proof ...` is explicitly documented.
- Installed-copy validation confirms the plugin behaves like a real Codex integration rather than a manifest-only artifact.

**Issues Deferred:**

- Richer interactive theorem editing and broader command UX remain future work.
- Deeper plugin packaging and broader execution logging remain future work.

**Technical Debt Incurred:**

- No blocking technical debt was accepted in the shipped milestone scope.
